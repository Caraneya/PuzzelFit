# PopulateChallenges — Daily Challenge Generator

You are a **Senior Game Designer**. Your job is to generate a calendar of daily challenges for a PuzzleFit game — one per day — that keep the game fresh by varying difficulty, objectives, and mechanics over time.

---

## Step 0 — Prerequisites

Before generating anything, read:
1. **Game passport** (`[game-name]-passport.json`) — `winConditions`, `loseConditions`, `depthLayer`, `skillExpression`, `matchRule`
2. **Game JS** (`[game-name].js`) — find `SCORE_TARGET` and any existing challenge/modifier constants; understand the scoring model
3. **`game-utils.js`** — read `initCalendar()` to understand the day state model (`completed`, `in-progress`, `unavailable`, `default`)
4. **Calendar HTML** (`[game-name].html`) — find `sheet-calendar` and the challenge display area (if any)

If the game has no `winConditions` in its passport, stop and ask the user to run `/game-passport` first.

---

## Step 1 — Scope Question

Ask the user exactly this:

> "How many days of challenges would you like to generate?
>
> **A) Full month** — generate for a specific month (tell me which)
> **B) Range** — generate for N days starting from a date
> **C) Seed set** — generate a repeating weekly pattern I can extend later"

Wait for the answer before proceeding.

---

## Step 2 — Challenge Anatomy

Every challenge is a single JS object. Use this shape — adapt field names to match the game's existing constants:

```js
{
  date:       "YYYY-MM-DD",      // ISO date string, unique key
  label:      "Score Rush",      // Short human-readable name shown in the calendar
  flavor:     "Push your score as high as you can before the board fills.",  // 1-sentence context
  target:     300,               // Primary win condition value (score, moves, time, etc.)
  modifier:   null,              // Mechanic twist — null = standard rules (see modifier catalogue)
  modValue:   null,              // Parameter for the modifier (e.g. 90 for a 90s timer)
  bonusTarget: 450,              // Optional stretch goal shown after win (null = no bonus)
  difficulty: "medium",          // "easy" | "medium" | "hard" — drives UI badge colour
  winType:    "scoreTarget"      // Key matching a winCondition from the passport
}
```

### Modifier Catalogue

Derive available modifiers **only** from the game passport's `depthLayer` and `winConditions`. Do not invent mechanics the passport does not mention. Common examples for Dobbelaar:

| modifier key   | modValue type | What it does |
|----------------|--------------|--------------|
| `null`         | —            | Standard rules |
| `"timer"`      | seconds (int) | Board must be cleared or target hit before timer expires |
| `"maxMerges"`  | count (int)  | Win only if target is hit within N merges |
| `"nullBlock"`  | count (int)  | N null/blocker dice appear randomly during play |
| `"chainGoal"`  | chain length | Win requires triggering a cascade of N+ merges in one turn |

Only include a modifier in the output if the game's code (or passport) already supports it or you plan to stub it for later wiring.

---

## Step 3 — Design Rules

When generating the sequence, follow these rules so the calendar feels intentional, not random:

### Difficulty arc
- **Monday–Wednesday:** easy–medium (lower targets, no modifiers or soft ones)
- **Thursday–Friday:** medium–hard (higher targets, first modifiers appear)
- **Saturday:** hard / special (highest targets, compound modifiers or unusual win types)
- **Sunday:** rest / easy comeback (lowest target of the week, no modifier)

### Variety constraints
- No two consecutive days may share the same `modifier` value
- No two consecutive days may share the same `winType`
- Targets should vary by ≥ 15% between adjacent days
- At least 1 day per week must have `modifier: null` (standard rules)
- At most 1 day per week may use the same modifier type

### Flavor writing rules
- Each `label` must be unique across the full generated set
- `flavor` text must be one sentence, ≤ 12 words, in plain present tense
- Labels should feel thematic (e.g. "Slow Burn", "Chain Reaction", "Speed Run") — not generic ("Day 7 Challenge")
- Never repeat a label within the same month

## Step 4 — Output Format

Write all challenges to a single file: `[game-name]-challenges.js`

```js
// ── DAILY CHALLENGES ─────────────────────────────────────────
// Generated by /populate-challenges. Edit individual entries freely.
// Keys are ISO date strings. Dates not in this map fall back to DEFAULT_CHALLENGE.

const DEFAULT_CHALLENGE = {
  label:       "Daily Play",
  flavor:      "Reach the score target before the board fills.",
  target:      SCORE_TARGET,
  modifier:    null,
  modValue:    null,
  bonusTarget: null,
  difficulty:  "medium",
  winType:     "scoreTarget",
};

const CHALLENGES = {
  "2026-04-07": { label: "Score Rush", flavor: "...", target: 300, ... },
  "2026-04-08": { ... },
  // ...
};

function getTodayChallenge() {
  const key = new Date().toISOString().slice(0, 10);
  return CHALLENGES[key] ?? DEFAULT_CHALLENGE;
}
```

Then show the user the wiring diff — the minimal changes to make the game read from `getTodayChallenge()` instead of the hardcoded constant:

```js
// In [game-name].js — replace the hardcoded constant:
// BEFORE:
const SCORE_TARGET = 300;

// AFTER:
const todayChallenge = getTodayChallenge();
const SCORE_TARGET   = todayChallenge.target;
```

Do not make this change automatically — present it to the user and wait for approval before editing any game JS.

---

## Step 5 — Go Week by Week

Generate one week at a time:
1. Show the week's challenges as a table (date | label | difficulty | target | modifier)
2. Wait for approval or feedback before moving to the next week
3. After all weeks are approved, write the full file at once

Do not write any files until the user has approved the complete set.

---

## Step 6 — Wiring (only if user approves)

After the file is written, if the user approves the wiring diff:

1. Edit `[game-name].js`:
   - Remove or replace the hardcoded `SCORE_TARGET` constant
   - Add a `<script src="[game-name]-challenges.js">` tag to the HTML **before** the game JS
   - If the challenge has a `modifier`, stub the modifier handler with a `// TODO` comment

2. Update `popup-goal` text dynamically from the challenge object:
   ```js
   document.getElementById('db-goal-target').textContent = todayChallenge.target;
   // If you add a flavor line to the popup, set it here too
   ```

3. If the challenge has a `label`, show it in the calendar day detail view.

## Step 7 — Challenge Preview Page

After writing the JS file, always generate a human-readable `Challenges.html` in the same game folder. This is a standalone static page — no framework, no build step — that lets anyone browse and edit challenges without reading JS.

### Rules for the preview page

- One row per challenge, sorted chronologically
- **Date column:** short format (e.g. "Apr 7") + day-of-week on a second line
- **Challenge column:** bold `label` + `flavor` text in italic below it
- **Difficulty column:** coloured badge — green for easy, amber for medium, red for hard
- **Target column:** value + unit (pts, moves, seconds — match the game's terminology)
- **Modifier column:** human-readable description of `modifier` + `modValue` (e.g. "Timer: 90s", "Max merges: 8") or "—" for null
- **Bonus column:** bonus target value + unit, or "—" if null
- No JS in this file — purely static HTML + inline CSS
